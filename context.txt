# Technical Architecture Context: Unity Quiz Game

## 1. System Architecture Overview

The system relies on a decoupled, state-preserving architecture using standard Unity components, C# collections, and data serialization. The core gameplay loop runs via a scene-reloading lifecycle where data persistence is decoupled from Unity's standard Monobehaviour destruction using static memory allocations.

* 
**Game Manager Pattern:** A single persistent engine driven by a dedicated orchestrator (`GameManager`) handles state progression, random selection, user evaluation, and scene transits.


* 
**Data Serialization Class:** Custom structural definitions (`Question`) are marked with serialization metadata to hook cleanly into the Unity Engine Editor workflow.



---

## 2. Component Blueprint & Data Structures

### Class: `Question`

A lightweight, non-Monobehaviour data container class representing individual puzzle/quiz parameters.

* **Attributes:**
* 
`[System.Serializable]`: Allows nested properties to be exposed directly inside the Unity Inspector.


* 
`public string fact`: Stores the text/statement queried to the user.


* 
`public bool isTrue`: Solution boolean mapping whether the fact is explicitly correct or false.





### Class: `GameManager`

A Monobehaviour coordinator script bound to an empty `GameManager` GameObject in the scene hierarchy.

* **Data Fields:**
* 
`public Question[] questions`: A static-capacity Master Array populated directly inside the Unity Inspector containing the full question database.


* `private static List<Question> unansweredQuestions`: A globally tracked, dynamic collection representing the active matching pool. Being marked `static` guarantees memory allocation persists cleanly across scene resets and loads.


* 
`private Question currentQuestion`: Tracks the active single instance evaluation parameters.





---

## 3. Core Engine Mechanics & Logic Flow

### State Initialization (`Start()`)

1. Executed immediately upon scene compilation, boot, or programmatic iteration loops.


2. Evaluates the active database state. If `unansweredQuestions` is unassigned (`null`) OR if the tracking query total is spent (`unansweredQuestions.Count == 0`), it initializes/re-initializes the active run state.


3. Clones elements out of the master configuration array into the dynamic pool allocation using the `System.Linq` namespace:
```csharp
unansweredQuestions = questions.ToList();

```


4. Proceeds to call the question selection routine (`GetRandomQuestion()`).



### Selection Engine (`GetRandomQuestion()`)

1. Generates an index bounds variable utilizing random integer constraints:
```csharp
int randomQuestionIndex = Random.Range(0, unansweredQuestions.Count);

```


2. Resolves the memory reference to `currentQuestion` using the determined index.


3. Dispatches updates to structural UI text nodes to mirror the value of `currentQuestion.fact`.

### Validation State & Routing Mechanics

Two dedicated wrapper methods execute upon corresponding UI click events:

* `UserSelectTrue()`
* `UserSelectFalse()`

**Internal Processing Loop:**

1. Evaluates incoming selection states explicitly against `currentQuestion.isTrue`.
2. Registers results immediately via logging interfaces or UI state animations.
3. Pops the evaluated entry from the active list to prevent repeating questions:
```csharp
unansweredQuestions.Remove(currentQuestion);

```


4. Triggers an asynchronous or instant scene reload cycle through the Unity Scene Management suite to refresh text layout bindings and update elements for the subsequent frame loop:
```csharp
SceneManager.LoadScene(SceneManager.GetActiveScene().buildIndex);

```

## 4. Scene Interface & Layout Setup (Unity UI)

* 
**Render Pipeline:** Set to a dedicated 2D baseline project orientation operating out of a unified parent Canvas container.


* 
**Background Camera Config:** Configured with a dark backdrop preset (`Basic Back Dark`) to enhance visual hierarchy.


* 
**`QuestionPanel` Component:** Anchored to scale natively across the horizontal screen span while adhering strictly to top-bounding vertical anchors.


* 
**Interaction Targets (Buttons):** * `TrueButton` (Saturated blue styling, standard text asset reads "TRUE").


* 
`FalseButton` (Saturated red styling, standard text asset reads "FALSE").


* 
**Dynamic UI Layout:** Responsive anchoring setup ensures scaling behaviors are isolated symmetrically across half of the display canvas width. Hover transformations dynamically transition drop-shadow distances and tint saturation layers to reinforce interaction feedback.
